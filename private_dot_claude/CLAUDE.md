# Personal Preferences

## Communication Style

- Be concise. Lead with the answer, not the reasoning.
- Skip preamble, filler words, and trailing summaries.
- Avoid bullets in the "What" and "Why" sections of PR descriptions if possible.

## Code Style

- Prefer simple, direct solutions over clever or over-engineered ones.
- Don't add comments, docstrings, or type annotations to code I didn't ask you to change.
- Don't add error handling or validation beyond what's needed for the current task.
- Don't refactor surrounding code when fixing a bug or adding a feature.

## Dev environments

- Repos with devbox.json config files will usually need to run shell commands via devbox, i.e. `devbox yarn`, etc.

## Git

- Create focused, atomic commits with clear messages.

## Behavior

- Ask before taking irreversible or high-blast-radius actions.
- Don't retry a failing command more than once without diagnosing the root cause.
- When blocked, ask rather than brute-forcing past safety checks.
